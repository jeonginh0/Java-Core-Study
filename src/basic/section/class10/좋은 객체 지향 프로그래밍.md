# 좋은 객체 지향 프로그래밍?
- 객체 지향 프로그래밍은 컴퓨터 프로그램을 명령어의 목록으로 보는 시각에서 벗어나 여러개의 독립된 단위, 즉 "객체"들의 모임으로 파악하고자 하는 것.
- 각각의 객체는 메시지를 주고받고, 데이터를 처리할 수 있다(협력)
- 객체 지향 프로그래밍은 프로그램을 유연하고 변경이 용이하게 만들기 때문에 대규모 소프트웨어 개발에 많이 사용된다.

프로그래밍 언어에서 궁극의 유연함, 궁극의 변경의 용이성이 적용되도록 설게할 수 있는 특성이 있다.
바로 '다형성(Polymorphism)'이다.

## 다형성 실세계 비유
- 실세계와 객체 지향을 1:1로 매칭할 순 없음
- 그래도 실세계의 비유로 이해하기에는 좋음
- 역할과 구현으로 세상을 구분

### 운전자 - 자동차
운전자는 자동차를 K3를 타다가 아반떼로 바꿔서 탄다. 그대로 운전할 수 있다.
운전자라는 역할입장에서 자동차를 바꿔도 운전자에게 영향을 주지는 않는다.
자동차라는 역할이 있기 때문에 '유연하고 변경이 용이'하다고 하는것이다.

자동차 역할과 구현으로 분리되는 이유는 자동차가 바뀌더라도 운전자 역할은 바뀌지 않기 때문임.
클라이언트가 자동차의 내부 구조를 다 알 필요는 없음.

### 공연 무대 - 로미오와 줄리엣 공연
공연할 때 배우는 대체가능하다. 로미오, 줄리엣이라는 개념을 역할로 구분해야 유연한 대체 가능성을 가질 수 있다.

변경이 클라이언트 역할에 영향을 주지 않는다. 김태희가 하든 송혜교가 하든 줄리엣 역할 자체는 바뀌지 않는다.

### 예시
- 운전자 - 자동차
- 공연 무대
- 키보드, 마우스, 세상의 표준 인터페이스들
- 정렬 알고리즘
- 할인 정책 로직

## 역할과 구현을 분리
- 역할과 구현으로 구분하면 세상이 단순해지고, 유연해지며 변경도 편리해진다.
- 장점
  - 클라이언트는 대상의 역할(인터페이스)만 알면 된다.
  - 클라이언트는 구현 대상의 내부 구조를 몰라도 된다.
  - 클라이언트는 구현 대상의 내부 구조가 변경되어도 영향을 받지 않는다.
  - 클라이언트는 구현 대상 자체를 변경해도 영향을 받지 않는다.
- 자바 언어의 다형성을 활용
  - 역할 = 인터페이스
  - 구현 = 인터페이스를 구현한 클래스, 구현 객체
- 객체 설계 시(인터페이스)을 먼저 부여하고, 그 역할을 수행하는 구현 객체 만들기

## 객체의 협력이라는 관계부터 생각
- 혼자 있는 객체는 없다.
- 클라이언트: 요청, 서버: 응답
- 수 많은 객체 클라이언트와 객체 서버는 서로 협력 관계를 가진다.

## 다형성의 본질
- 인터페이스를 구현한 객체 인스턴스를 실행 시점에 유연하게 변경할 수 있다.
- 다형성의 본질을 이해하려면 협력이라는 객체사이의 관계에서 시작해야 함.
- 클라이언트를 변경하지 않고, 서버의 구현 기능을 유연하게 변경할 수 있다.

# 정리
- 다형성이 가장 중요하다!
- 디자인 패턴 대부분은 다형성을 활용하는 것이다.
- 스프링의 핵심인 제어의 역전(IoC), 의존관계 주입(DI)도 결국 다형성을 활용하는 것이다.
- 다형성을 잘 활용하면 마치 레고 블럭 조립하듯이! 공연 무대의 배우를 선택하듯이! 구현을 편리하게 변경할 수 있다.
